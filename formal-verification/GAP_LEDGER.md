verification artifacts. "Closed" means the repository now contains a checked
Rocq proof or a TLC-checked bounded model. "Scoped" means the claim is
intentionally narrower than the production system.

## Scoped Claims

| Claim | Scope |
|-------|-------|
| Byzantine fault handling | Storage/WAL records may be dropped, duplicated, or corrupted, and the bounded quorum model covers one-Byzantine HotStuff/PBFT-style safety. The model does not claim production Byzantine networking, liveness, malicious CPU execution, compromised cryptography, or arbitrary Rust memory corruption. |
| Certified compilation | The checked claim is for the Rocq reference map interface and proof-carrying trace checker boundary. It does not certify the Rust binary or LLVM/codegen pipeline. |
| Structural preservation | The sound target is successful checked construction. Raw total rebuilds can exceed bucket capacity and are not claimed to preserve structure for every input. |
| Persistent root descriptor/reopen | The checked claim covers descriptor publication, root-kind/final-flag/empty-payload validation, arena-count bounds, root-load failure fallback, checkpoint skip-threshold selection, and lazy-read fail-closed semantics. It does not claim recovery of checkpoint-only state after descriptor corruption once older WAL records have been intentionally truncated, nor does it prove kernel/filesystem internals below the modeled storage boundary. |
| Persistent lazy mutation | The checked claim covers public mutation preflight and WAL append ordering around lazy-load errors and no-op mutations. It does not prove arbitrary concurrent mutation schedules, kernel write ordering below the WAL abstraction, or correctness after pre-existing checkpoint corruption. |
| Persistent WAL write atomicity | The checked claim covers byte/char value serialization, preflight, WAL append, atomic write no-op behavior, replay correspondence, and document commit ordering at the WAL abstraction boundary. It does not prove nondeterministic serializers, arbitrary concurrent writer interleavings, kernel write ordering below successful WAL append/sync, or certified Rust compilation. |
| Persistent vocab WAL atomicity | The checked claim covers public vocabulary insert, manual-index insert, and batch insert WAL-before-visible-mutation ordering, duplicate stability, reindex/collision rejection, exact reverse-index membership, and replay correspondence. It does not prove arbitrary concurrent writer interleavings, kernel write ordering below successful WAL append/sync, or certified Rust compilation. |
| Concurrent checkpoint/public vocab APIs | The checked claim covers `ConcurrentVocabARTrie` queue and lock-free inserts racing with public `checkpoint()`/`flush()`: checkpoint takes an exclusive publication gate, inserts take shared publication guards, duplicate queue batches stabilize to one index, public operation histories have a sequential vocabulary-map explanation, and checkpoint/flush publication reopens with the returned indices. It does not claim per-insert crash durability for lock-free inserts that have not reached checkpoint/flush, arbitrary mutation APIs outside this wrapper, kernel/filesystem internals below successful sync, wait-free progress during checkpoint, or certified Rust/LLVM compilation. |

| Recovery planner durable prefix | The checked claim covers missing/clean/corrupt recovery mode selection, trusted-root/checkpoint skip-threshold selection, retained archive/pending/active replay order, and first-corruption durable-prefix replay for byte and char recovery.
